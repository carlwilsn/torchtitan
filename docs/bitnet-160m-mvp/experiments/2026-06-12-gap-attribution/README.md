## The question

The 1700-step seed-locked MVP comparison
([`../../results/README.md`](../../results/README.md),
[`../../results/results_summary.txt`](../../results/results_summary.txt)) found:

| Quantity (step 1700) | Stock | BitNet | Gap |
| --- | ---: | ---: | ---: |
| Train loss | 1.4123 | 1.5438 | **+0.1314 nats** |
| Val loss | 1.4342 | 1.5576 | **+0.1233 nats** |
| Val perplexity | 4.1965 | 4.7473 | +0.55 |

That run *measured* the gap but didn't *explain* it. BitNet b1.58 differs from stock
in several stacked ways ‚Äî ternary (absmean) weight quantization with STE, per-token
int8 absmax activation quantization with STE, and the extra SubLN-style pre-RMSNorm
inside each BitLinear. Any of them (or their training-dynamics interaction) could own
the +0.13 nats. This experiment decomposes the gap by ablating one component at a time.

## Design ‚Äî controlled ablation ladder

All variants share the `llama3_160m` base config: **same seed lock (`debug.seed=42`),
same model shape, same data order, same optimizer/schedule, same lm_head filter**
(the output projection stays a stock Linear in every BitNet-family variant, so "FP16
head" is already the baseline, not a separate arm). The only degree of freedom is
which quantizers run inside BitLinear's forward. Configs live in
[`torchtitan/models/llama3/config_registry.py`](../../../../torchtitan/models/llama3/config_registry.py).

| Variant (CONFIG name) | BitLinear structure (pre-norm) | Ternary weights (STE) | Int8 activations (STE) | What its gap-to-stock isolates |
| --- | :-: | :-: | :-: | --- |
| `llama3_160m` (stock) | ‚Äî | ‚Äî | ‚Äî | baseline |
| `llama3_160m_bitnet_structure_only` | ‚úì | ‚Äî | ‚Äî | pure structural/optimization effect of the extra norms |
| `llama3_160m_bitnet_fp16_weights` | ‚úì | ‚Äî | ‚úì | activation quantization cost (weights stay FP) |
| `llama3_160m_bitnet_no_actquant` | ‚úì | ‚úì | ‚Äî | weight ternarization cost (activations stay FP) |
| `llama3_160m_bitnet` (full) | ‚úì | ‚úì | ‚úì | the full +0.13 nat gap, reproduced on the short config |

## PREDICTION (committed before results)

From the goal log (`.vault-chat/supervisor/goals/bitnet-160m-gap-attribution-test.md`),
recorded before any run started:

> If the quality gap is mostly from **weight ternarization**, then "BitLinear
> structure but FP16 weights" should sit **close to stock**, while true ternary
> BitNet keeps most of the loss gap. If the gap is mostly from **training
> dynamics / STE / scaling**, the FP16-weights ablation may **still underperform**
> stock.

Decomposed into per-variant expectations:

| Variant | Predicted position |
| --- | --- |
| `structure_only` | ‚âà stock (extra RMSNorms alone shouldn't cost much; if this one shows a gap, the "gap" is partly structural/optimization, not quantization) |
| `fp16_weights` | near stock if weights own the gap; meaningfully below stock if activation-quant STE / dynamics own it |
| `no_actquant` | carries most of the gap if weights own it; near stock if activations own it |
| full `bitnet` | reproduces a gap of the same sign as the MVP (magnitude may differ on the short config) |
